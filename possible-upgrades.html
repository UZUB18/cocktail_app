Current app surface

What the app already does well

The existing code is concentrated in src/App.tsx, with persistent state in src/storage.ts and the catalog in src/data/catalog.ts. The strongest upgrade path is to keep the local-first workflow and make the user data more useful, portable, and service-ready.

Library

Searchable cocktail catalog

Users can search by cocktail name, family, base spirit, variations, and ingredients across the large static catalog.

Recipe workspace

Detailed specs and tabs

Each drink has ingredients, method, flavor notes, substitutions, variations, and batching guidance.

Home bar

Inventory and shopping list

The app compares recipe ingredients against stored inventory and surfaces drinks that are nearly makeable.

Service

Batch planner

The current calculator already handles servings, dilution, estimated ABV, bottle counts, cautions, and prep checklist state.

Upgrade concepts

High-value features to consider next

Filter the concepts by product area. Priority reflects implementation value for this codebase, not just feature size.

Data safety P0 Now

Export, import, and reset center

Local storage is useful but fragile. Give users a real control panel for backing up saved recipes, notes, inventory, and collections.

  • Download local DB as JSON
  • Import and validate a backup file
  • Show last backup date and object counts
Low riskImmediate trustStorage
Real DB path P1 Next

Optional cloud sync

Keep offline-first localStorage, but add an optional account-backed sync layer for users who switch browsers or devices.

  • SQLite or Supabase-backed profile state
  • Conflict handling by updated timestamp
  • Local fallback when offline
AuthSyncDatabase
Search P0 Now

Faceted search and sorting

Search currently uses one broad text match. Add filters that match how people choose drinks: spirit, family, difficulty, prep time, glassware, and makeable status.

  • Filter chips above results
  • Sort by makeable, saved, or fast prep
  • Saved search presets for party planning
CatalogDiscoveryFast win
Inventory P0 Now

Ingredient intelligence

The current ingredient matcher is text-based. A normalized ingredient model would understand aliases, categories, substitutions, bottle depletion, and freshness windows.

  • Alias map for lime juice, fresh lime, lime
  • Stock levels and low-stock thresholds
  • Substitution-aware makeable scores
Home barMatchingData model
Planning P1 Next

Party mode and menu builder

Extend saved specs and the batch planner into a dedicated event workflow for building a short menu, generating a shopping run, and estimating prep time.

  • Pick 3 to 6 menu drinks
  • Aggregate all ingredients by serving count
  • Generate prep labels and service timeline
EventsBatchingShopping
Recipe editor P0 Now

Structured custom recipe editor

The quick-add flow only captures a small draft. A proper editor would reduce bad data and make custom drinks feel first-class.

  • Ingredient rows with amount, unit, name
  • Flavor meter controls and method steps
  • Duplicate detection before saving
Custom recipesValidationEditor
Tasting P1 Next

Tasting log and adjustment history

Notes exist today, but they are free text. Turn troubleshooting and notes into a history of measured changes and outcomes.

  • Log issue, adjustment, result, rating
  • Compare versions of a custom spec
  • Suggest fixes from past outcomes
NotesIterationFlavor
Responsive UX P0 Now

Persistent mobile navigation

The current sidebar is hover-driven on desktop. Mobile would benefit from a bottom nav or command sheet so core views are always reachable.

  • Bottom tabs for Home, Search, Inventory, Batch
  • Command palette for faster desktop navigation
  • Keyboard shortcut labels only where functional
MobileNavigationAccessibility
App foundation P2 Later

Split the large app module

The main file carries views, helpers, state, and calculations. Splitting modules would make future database and editor work easier to maintain.

  • Move batch math to a domain module
  • Move view components into feature folders
  • Centralize catalog and storage adapters

Practical order

A sane rollout sequence

This order starts with data safety and UX improvements, then moves into deeper product logic and architecture work.

Phase 1

Make local data trustworthy

  • Export and import JSON backups
  • Reset confirmation with storage preview
  • Recipe editor validation
  • Faceted search chips
Phase 2

Make the bar workflow smarter

  • Ingredient alias database
  • Makeable score and substitution hints
  • Party menu builder
  • Aggregated shopping list
Phase 3

Make it portable and scalable

  • Optional synced account state
  • PWA install and offline cache
  • Module split by feature area
  • Versioned data migration layer
Implementation map

Where the work would land

The current repo does not include a backend database. These notes point to the local files that would change first.

Storage

Version the local DB

Add schema versioning, migrations, backup metadata, and import validation around the existing localStorage state.

  • Keep the current default seed state
  • Guard imported objects before writing
  • Prepare a future sync adapter interface
src/storage.ts
Feature split

Move views into feature folders

Extract Home, Search, Recipe, Inventory, Batch, Collections, and Troubleshoot views from the monolithic app file.

  • Preserve existing behavior first
  • Move helper functions after coverage or manual QA exists
  • Keep shared UI primitives small
src/App.tsx
Catalog

Add ingredient taxonomy

Create a normalized ingredient layer without rewriting all 130 catalog entries at once.

  • Ingredient aliases and categories
  • Optional canonical ID per ingredient
  • Substitution and inventory scoring helpers
src/data/catalog.ts
Interface

Upgrade navigation and mobile density

Keep the dark app style, but make primary actions easier to reach on touch devices and small screens.

  • Bottom navigation at mobile widths
  • Command palette for desktop
  • More stable spacing around dense controls
